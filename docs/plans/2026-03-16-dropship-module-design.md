# 代采代发模块设计文档

> 状态: 已确认 | 日期: 2026-03-16

## 一、业务背景

代采代发业务：从上游供应商采购，再发给下游分散客户。每天约十几单采购+十几单发货，SKU 多为一次性非常规商品。

**现状痛点：** 每单需在 ERP 里操作十几步（建产品→建采购单→审核→入库→建销售单→出库→发货），一次性 SKU 污染常规产品库和报表。

**目标效果：** 业务员填一张表单（1分钟），财务在付款工作台批量确认（每天5分钟），系统自动生成全套财务单据，操作量砍掉 80%。

---

## 二、数据模型

### 2.1 核心模型：DropshipOrder（独立新模型）

不复用现有 Order / PurchaseOrder，与常规业务完全解耦。

```
DropshipOrder
├── ds_no: 单号（自动生成，格式 DS{YYMMDD}-{序号}）
├── account_set: FK → AccountSet（必填）
├── status: draft | pending_payment | paid_pending_ship | shipped | completed | cancelled
│
├── # 采购信息
├── supplier: FK → Supplier
├── product: FK → Product
├── product_name: CharField（录入时的商品名，用于快速新建）
├── purchase_price: Decimal（采购单价，含税）
├── quantity: Int
├── purchase_total: Decimal（采购总额，自动计算）
├── invoice_type: special | normal（专票/普票）
├── purchase_tax_rate: Decimal（专票默认13%，普票不适用）
│
├── # 销售信息
├── customer: FK → Customer
├── platform_order_no: CharField（平台合约订单编号，必填）
├── sale_price: Decimal（销售单价，含税）
├── sale_total: Decimal（销售总额，自动计算）
├── sale_tax_rate: Decimal（默认13%）
├── settlement_type: prepaid | credit（先款后货/赊销）
├── advance_receipt: FK → ReceiptBill（nullable，先款后货时关联预收款）
│
├── # 毛利（拆税后实时计算）
├── gross_profit: Decimal
├── gross_margin: Decimal（毛利率%）
│
├── # 物流信息
├── shipping_mode: direct | transit（供应商直发/过手转发）
├── carrier_code: CharField（nullable，快递公司代码）
├── carrier_name: CharField（nullable，快递公司名称）
├── tracking_no: CharField（nullable，快递单号）
├── kd100_subscribed: Boolean（默认false）
├── last_tracking_info: TextField（nullable）
│
├── # 状态管理
├── urged_at: DatetimeField（nullable，催付款时间）
├── cancel_reason: CharField（nullable）
├── note: TextField（nullable，备注）
│
├── # 关联单据（系统自动生成后回填）
├── payable_bill: FK → PayableBill（nullable）
├── disbursement_bill: FK → DisbursementBill（nullable）
├── receivable_bill: FK → ReceivableBill（nullable）
│
├── # 付款信息
├── payment_method: CharField（nullable，银行转账/冲减借支）
├── payment_employee: FK → Employee（nullable，冲减借支时选择的员工）
│
├── creator: FK → User
├── created_at, updated_at
```

### 2.2 现有模型扩展

**ReceivableBill 新增字段：**
- `platform_order_no`: CharField（nullable，从 DropshipOrder 推送过来）
- `dropship_order`: FK → DropshipOrder（nullable，用于发票确认时识别来源）

**Product：**
- 代采代发新建的产品 `category='代采代发'`，与常规产品隔离

**DisbursementMethod（付款方式配置表）：**
- 新增 `冲减借支` 付款方式

### 2.3 暂不新增员工借支余额模型

冲减借支的凭证通过 `其他应收款` 科目 + 员工辅助核算跟踪，不单独建借支余额表。员工借支余额通过查询 GL 辅助余额获取。

## 三、状态流转

```
草稿 ──→ 待付款 ──→ 已付待发 ──→ 已发货 ──→ 已完成
 │          │          │
 └──→ 已取消 ←─────────┘
```

| 状态 | 触发方式 | 操作人 | 系统自动操作 |
|------|---------|--------|------------|
| 草稿→待付款 | 业务员点"提交" | 业务员 | 建应付单；如需新建产品(category=代采代发) |
| 待付款→已付待发 | 财务在工作台确认付款 | 财务 | 建付款单；生成凭证A |
| 已付待发→已发货 | 业务员填快递单号确认发货 | 业务员 | 建应收单(含platform_order_no)；过手模式建出入库单；订阅快递100 |
| 已发货→已完成 | 快递100签收回调 | 系统自动 | 标记完成 |
| 任意→已取消 | 业务员/财务取消 | 业务员/财务 | 冲销已生成单据+红冲凭证（见第六节） |

## 四、凭证方案

### 4.1 凭证A — 财务确认付款时生成

**按付款方式区分贷方科目，同一供应商批量合并为一张凭证。**

摘要格式：`代采代发付款 {供应商名} {DS单号1}/{DS单号2}`

| 付款方式 | 借方 | 贷方 |
|---------|------|------|
| 银行转账 | 预付账款-供应商（辅助:供应商） | 银行存款 |
| 冲减借支 | 预付账款-供应商（辅助:供应商） | 其他应收款-员工借支（辅助:员工） |

- 凭证类型：`付`
- 付款单每单独立，凭证按供应商合并

### 4.2 凭证B — 确认销售发票时生成

**在发票管理模块操作。从应收单推发票→确认发票时，系统检测来源为代采代发，额外生成成本结转分录。**

**销项税始终拆出（13%），进项税仅专票。**

以采购价 6,800、售价 7,200 为例：

**专票场景：**

| 摘要 | 科目 | 借方 | 贷方 |
|------|------|-----:|-----:|
| 代采代发收入 DS0315-01 | 应收账款-客户 | 7,200.00 | |
| | 应交税费-销项税额 | | 828.32 |
| | 主营业务收入-代采 | | 6,371.68 |
| 代采代发成本 DS0315-01 | 应交税费-进项税额 | 782.30 | |
| | 主营业务成本-代采 | 6,017.70 | |
| | 预付账款-供应商 | | 6,800.00 |

**普票场景（无进项抵扣，含税全额入成本）：**

| 摘要 | 科目 | 借方 | 贷方 |
|------|------|-----:|-----:|
| 代采代发收入 DS0315-01 | 应收账款-客户 | 7,200.00 | |
| | 应交税费-销项税额 | | 828.32 |
| | 主营业务收入-代采 | | 6,371.68 |
| 代采代发成本 DS0315-01 | 主营业务成本-代采 | 6,800.00 | |
| | 预付账款-供应商 | | 6,800.00 |

- 凭证类型：`记`
- 触发链路：ReceivableBill → Invoice (push from receivable) → confirm → 检查 dropship_order_id → 生成含成本的合并凭证

### 4.3 凭证C — 客户回款时生成

**在应收管理模块统一操作，与常规业务回款流程一致。**

| 摘要 | 科目 | 借方 | 贷方 |
|------|------|-----:|-----:|
| 代采代发回款 DS0315-01 | 银行存款 | 7,200.00 | |
| | 应收账款-客户 | | 7,200.00 |

- 凭证类型：`收`

### 4.4 凭证链路总览

```
时点          凭证    借方              贷方
────          ────    ────              ────
付款          A       预付账款          银行存款/其他应收款(借支)
开票确认      B       应收+进项+成本    销项+收入+预付账款
客户回款      C       银行存款          应收账款
```

### 4.5 会计科目要求

| 科目 | 编码 | 说明 |
|------|------|------|
| 预付账款 | 1123 | 付款后挂预付，开票时转成本 |
| 应付账款 | 2202 | 提交时记录负债 |
| 应收账款 | 1122 | 发货时记录债权 |
| 银行存款 | 1002 | 实际收付款 |
| 其他应收款 | 1221 | 员工借支，辅助核算到人 |
| 主营业务收入-代采 | 6001-XX | 独立二级科目 |
| 主营业务成本-代采 | 6401-XX | 独立二级科目 |
| 应交税费-销项税额 | 2221-02 | 销售增值税 |
| 应交税费-进项税额 | 2221-01 | 采购增值税（仅专票） |

## 六、财务付款工作台

### 6.1 界面布局

- 按供应商分组显示所有"待付款"状态的代采代发单
- 每单显示：商品、采购额、售价、毛利、客户回款状态（✅已收款/⏳赊销）、催付款标记（"催"）
- 顶部汇总：今日待付笔数、合计金额、已预付笔数
- 批量勾选 → 选付款方式 → 确认付款

### 6.2 付款方式

| 方式 | 操作 | 凭证贷方 |
|------|------|---------|
| 银行转账 | 选择付款方式即可 | 银行存款 |
| 冲减借支 | 额外出现员工下拉框 | 其他应收款-员工借支 |

### 6.3 批量付款规则

- 可跨供应商勾选
- 付款单：每单独立一张 DisbursementBill
- 凭证：同一供应商的合并为一张凭证，摘要含公司名+所有单号
- 不同供应商的分别生成各自的凭证

## 七、取消流程

| 取消时点 | 冲销操作 |
|---------|---------|
| 待付款阶段 | 冲销应付单 |
| 已付待发阶段 | 冲销应付单 + 红冲凭证A + 建退款单（银行转账→退银行；冲减借支→恢复借支余额） |
| 已发货后 | v1 不支持取消 |

## 八、数据隔离策略

| 维度 | 隔离方式 |
|------|---------|
| 订单 | 独立 DropshipOrder 模型，不进入 Order/PurchaseOrder 表 |
| 产品 | 新建产品 category='代采代发'，产品列表可按分类过滤 |
| 库存 | 过手模式走专用"代采代发中转仓"（虚拟仓） |
| 会计科目 | 独立二级科目：主营业务收入-代采、主营业务成本-代采 |
| 应收应付 | 正常走 AR/AP 流程，ReceivableBill 标记 dropship_order_id 可区分来源 |

## 九、两种发货模式

### 供应商直发
- 货不经过我们仓库
- 不产生出入库单
- 凭证中无库存商品科目

### 过手转发
- 货经过代采代发中转仓（虚拟仓）
- 发货时自动建：采购入库单（入中转仓）+ 销售出库单（出中转仓）
- 中转仓库存不影响常规仓库数据

## 十、报表需求

1. **代采代发月度汇总** — 按客户、按供应商两个维度
2. **代采代发毛利报表** — 每单毛利、平均毛利率、按时间趋势
3. **代采代发应收未收汇总** — 赊销客户的未回款明细

## 十二、与现有系统的集成点

| 集成点 | 说明 |
|-------|------|
| 发票管理 | 从 ReceivableBill 推发票 → 确认时检查 dropship_order_id → 生成含成本的凭证B |
| 应收管理 | 赊销回款在应收管理统一操作，流程与常规一致 |
| 快递100 | 复用现有 KD100 集成，订阅物流→签收自动完成 |
| 供应商管理 | 快速新建 + 批量导入，复用现有 Supplier 模型 |
| 产品管理 | 复用现有 Product 模型，category 标记隔离 |
| 客户管理 | 复用现有 Customer 模型 |
| 付款方式 | DisbursementMethod 新增"冲减借支" |
